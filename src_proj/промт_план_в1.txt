**Цель:** Создать детализированный пошаговый план разработки проекта на основе предоставленной архитектуры.  

---

## **Контекст проекта**  
Разрабатывается веб-приложение для автоматической генерации краткого содержания (саммари) текстовых документов.  
**Ключевые параметры:**  
- **Входные данные:** Файлы форматов `doc`, `docx`, `txt`, `odt` (макс. 15 МБ).  
- **Язык:** Русский.  
- **Технологии:**  
  - База данных: MongoDB 8.2.1.  
  - Бэкенд: Python (FastAPI, `python-docx`, `transformers`, асинхронная работа с MongoDB).  
  - Фронтенд: React + TypeScript.  
  - ML-модель: Предобученная `mbart_ru_sum_gazeta` (адаптированная из проекта `gazeta`).  
- **Архитектура:**  
  - Многослойная структура (API, бизнес-логика, инфраструктура).  
  - Docker-контейнеризация для всех компонентов.  

---

## **Задача**  
На основе контекста сформируй **детальный план разработки**, разделенный на этапы по ответственности команд/ролей.  

**Требования к плану:**  
1. **Структура:**  
   - Разделение на три основные роли:  
     - **ML + База данных** (подготовка модели, настройка MongoDB).  
     - **Бэкенд** (API, бизнес-логика, интеграция с ML и БД).  
     - **Фронтенд** (UI/UX, интеграция с API).  
   - Для каждого этапа укажи:  
     - Конкретные задачи (например: "Настроить схемы MongoDB для сущностей Document и Summary").  
     - Зависимости от других этапов (например: "Требует завершения этапа ML").  
     - Критерии завершения (например: "Прохождение всех unit-тестов для парсинга файлов").  
     - Приоритет (высокий/средний/низкий).  

2. **Детализация:**  
   - Укажи порядок разработки (что должно быть сделано в первую очередь).  
   - Упомяни кросс-функциональные задачи (например: написание документации, настройка CI/CD).  
   - Добавь измеримые критерии успеха (например: "Время обработки файла ≤ 30 секунд", "ROUGE-L ≥ 0.4 для саммари").  

3. **Ограничения:**  
   - Не указывай действия с конкретными файлами (например: "Создать файл `mbart_gateway.py`").  
   - Избегай маркдауна в ответе — используй чистую структуру с подзаголовками и списками.  
   - Сохраняй фокус на архитектурных решениях, а не на коде.  

---

## **Пример ожидаемого формата плана**  
```
Этап 1: [Название этапа]  
Роль: [ML + БД / Бэкенд / Фронтенд]  
Задачи:  
- [Конкретная задача 1]  
- [Конкретная задача 2]  
Зависимости: [Этапы или компоненты, от которых зависит задача]  
Критерии завершения: [Измеримые результаты]  
Приоритет: [Высокий/Средний/Низкий]  

Этап 2: [...]  
...  
```  

--- 

**Важно:** План должен быть реализуемым в условиях реальной разработки, с акцентом на последовательность и минимизацию блокировок между командами. Учитывай ограничения (15 МБ на файл, поддержка только русского языка).
